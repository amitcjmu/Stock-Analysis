## Slide 16: Refinement Workflow

### **How We Prevent Mid-Sprint Surprises**

**Weekly Refinement Session:**
1. **Product team triages "Not Broken Down Yet"**
   - Break into smaller issues (1-2 day chunks)
   - Write acceptance criteria
   - Identify dependencies
2. **Tech lead identifies spikes**
   - "Do we know how to do this?"
   - If no: Create time-boxed spike (4-8 hours max)
   - If yes: Move directly to "Ready for Development"
3. **Dev team estimates effort**
   - 1-2 weeks, 3-4 weeks, etc.
   - Flagged if >5-8 weeks (needs further breakdown)
4. **Item moves to "Ready for Development"**
   - Sprint planning pulls from this column
   - No ambiguity, no mid-sprint pivots

**Result:**
- Sprint commitments honored 95% of the time
- Scope creep reduced by 80%
- Developer frustration ("changing requirements mid-sprint") eliminated

## Slide 17: Effort Estimation Standards

### **Standardized Sizing Across Teams**

**Effort Categories:**
- **1-2 weeks:** Small features, bug fixes, minor enhancements
- **3-4 weeks:** Medium features, integration work, refactoring
- **5-8 weeks:** Major features, new flow implementations
- **9-12 weeks:** Strategic initiatives (e.g., Decommission Flow)
- **12+ weeks:** Multi-quarter epics (e.g., Database Stabilization)

**Calibration:**
- Team calibrates estimates in retrospectives
- "1-2 weeks" means **one developer, two weeks** (not team effort)
- Items >5-8 weeks trigger automatic decomposition request

**Why This Matters:**
- Capacity planning accuracy
- Realistic roadmap commitments
- Early identification of under-scoped work
